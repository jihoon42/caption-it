# 캡션잇 (CAPTION-IT) — 접근성 자막 파이프라인 MCP

**자동 자막은 "있다"에서 멈춥니다 — 캡션잇은 "읽힌다"까지 갑니다.**
회의 녹음·강의·영상을 청각장애인 자막 규격(SDH)에 맞는 WebVTT/SRT로
결정론적으로 **생성·감사·보정**하는 MCP 서버입니다.

고장 난 자막이 들어오면, 근거 있는 위반 리포트와 내용 불변 보정본이 나옵니다
(`npm run demo` 출력 그대로):

```text
[BEFORE] 고장 자막                 [위반 리포트]                        [AFTER] 내용 불변 보정본
00:00:00.000 --> 00:00:01.000  →   cps_exceeded    42.5 CPS (≤14)   →   00:00:00.000 --> 00:00:00.416
이 자막은 일부러 규격을 어겨서       too_many_lines  3줄 (≤2줄)            이 자막은 일부러
읽기 속도와 줄 수 제한을            overlap         500ms 겹침            규격을 어겨서 읽기 속도와
동시에 위반하게 만들었습니다,       trailing_punct  줄 끝 쉼표             (표기·줄바꿈·타이밍만 수정,
                                   (규칙·수치·출처 근거 포함)              어휘 변경 0 — 축약이 필요한
                                                                        큐는 위반으로 정직하게 보고)
```

**누가 언제 쓰나**

- ⓐ **회의 녹음을 자막으로** — `transcribe_media`(STT) → `build_accessible_captions` → 화자 라벨·읽기 속도까지 맞춘 규격 자막
- ⓑ **유튜브에서 받은 자막 점검·보정** — `audit_captions`에 VTT/SRT를 넣으면 위반 리포트 + 내용 불변 자동 보정본
- ⓒ **링킷 프로필 연동** — 링킷(hearing-mcp) `support_band`를 그대로 전달하면 읽기 속도·노출 시간을 보수적으로 조정

**30초 시작** (STT 키·외부 네트워크 불필요)

```bash
npm ci && npm run build && npm run demo
```

**카카오톡 나벳방에서 말 걸어보기** — PlayMCP에 연결했다면 이렇게 시작하세요:

1. "회의 녹음을 자막으로 만들고 싶어. 키가 없으니 샘플로 먼저 보여줘." (샘플 전사 → 자막 생성)
2. "유튜브에서 받은 자막 파일이야. 접근성 규격에 맞는지 점검하고 고칠 수 있는 건 고쳐 줘." (감사·보정)
3. "전사 텍스트만 있고 타임스탬프는 없어. 10분짜리 강의인데 자막으로 만들어 줘." (근사 배분 경로)
4. "링킷에서 만든 내 소통 프로필로는 자막이 더 천천히 나왔으면 해." (support_band 연동)
5. "이 영상엔 박수 소리가 중요하고 중간에 노래도 나와. 자막에 표시해 줘." (sound_events·가사 ♪)

---

> **CAPTION-IT** = **CAPTION** + "**Caption It!**" — 링킷(LINKIT)의 자매 서버.
> 링킷이 소통이 성립할 조건을 만든다면, **캡션잇은 남겨진 소리를 읽히게 만듭니다.**

## 1. 왜 만들었나 — 자막이 "있다"와 "읽힌다"의 간극

자동 자막은 흔해졌지만, 청각장애인 기준의 자막 규격과는 거리가 멉니다.
화자가 누구인지 표시되지 않고, 소리 정보가 없고, 읽기 속도를 초과하는 큐가 쏟아집니다.
동시에 이 영역은 LLM에게 맡기면 조용히 틀리는 영역입니다 — 타임코드 산술, 읽기 속도(CPS) 계산,
줄당 자수 검증은 확률적 생성이 아니라 결정론적 계산의 일입니다.

캡션잇의 핵심 가치는 화려한 기능이 아니라, **공개 표준(Netflix Korean TTSG·방통위 가이드라인)의
수치를 룰북 데이터로 들고, 생성과 감사가 같은 룰북을 공유하는 신뢰 레이어**입니다.
에이전트는 "자막을 만들었다"가 아니라 "이 규격을 이 수치로 통과했다"를 말할 수 있게 됩니다.

## 2. 이 서버가 하지 않는 것 (Non-goals)

- **실시간 자막 처리를 하지 않습니다.** 레이턴시·스트리밍이 본질인 시스템은 request/response
  MCP의 폼팩터와 맞지 않습니다 — 즉, 링킷과 같은 기준에서 일관성을 지키기 위해 노력하였습니다.
  캡션잇이 맡는 것은 남겨진 기록의 비실시간 처리입니다.
- **텍스트를 축약·의역하지 않습니다.** 읽기 속도 위반이 표시 시간 연장으로 해결되지 않으면
  `needs_text_reduction`으로 보고할 뿐, 어휘를 줄이는 판단은 에이전트와 사용자의 몫입니다.
- **어떤 소리가 중요한지 판단하지 않습니다.** 소리 정보(`sound_events`)의 선택과 묘사는
  내용 판단이므로 클라이언트 LLM/사용자가 결정하고, 서버는 규격에 맞게 배치만 합니다.
- **오디오·전사를 저장하지 않습니다.** 모든 처리는 무저장(stateless)이며,
  프로필·세션·개인정보 저장소 자체가 없습니다.

## 3. 아키텍처 원칙

- **요약·의역·소리 묘사 = 클라이언트 LLM / 산술·규격 판정·형식 변환 = 서버(결정론).**
  같은 입력에는 같은 큐 분할, 같은 타임코드, 같은 위반 리포트가 나옵니다.
- **내용 불변 원칙.** 서버가 허용하는 텍스트 변경은 표기 규칙 수준(`...`→`…`, 줄 끝 마침표·쉼표
  제거, 공백 정규화)뿐이며, 전부 `fix_log`에 기록됩니다. 텍스트는 절대 잘려나가지 않습니다.
- **생성과 감사는 같은 룰북을 공유합니다.** `build_accessible_captions`가 만든 자막은
  서버 소관 규칙(타이밍·간격·겹침·줄 수·줄 길이·노출 시간)에서 `audit_captions`를
  무위반으로 통과하며, 텍스트 축약이 필요해 남는 위반(읽기 속도·줄 수 불가피분)은
  같은 큐의 `needs_attention`으로 반드시 선언됩니다 — 침묵 실패 없음
  (시드 고정 property 테스트 2,000케이스로 보증).
- **키가 없어도 동작합니다.** STT 프로바이더 미설정 시에도 자막 생성·감사·샘플 체험은
  완전 동작하며, 전사 요청에는 활성화 방법을 구조화해 안내합니다.

## 4. 도구 (4)

| 도구 | 역할 |
|---|---|
| `transcribe_media` | 공개 URL 음성/영상 → STT 전사 (타임스탬프 세그먼트). CLOVA/Azure/OpenAI 어댑터, 키 없이는 sample_id 체험 |

## 6. 링킷(hearing-mcp) 연동 — 자매 서버

링킷 `create_communication_profile` 응답의 `support_band`를 캡션잇의
`build_accessible_captions`/`audit_captions`에 그대로 전달하면:

- `intensive` → 해당 모드의 아동 등급 CPS 적용 + 최소 노출 1초
  (근거: 링킷 축 D(정보처리) — 전문용어·긴 문장에서 처리 부담이 크면 표시 시간 여유가 도움)

이 매핑은 의료적 판정이 아니라 "표시 시간을 얼마나 보수적으로 줄 것인가"이며,
링킷의 "상태는 에이전트가 든다" 설계와 같은 방식으로 인자 전달만으로 동작합니다.

## 7. STT 프로바이더 설정 (전부 선택)

| 프로바이더 | env | 비고 |
|---|---|---|
| CLOVA Speech | `CLOVA_SPEECH_INVOKE_URL`, `CLOVA_SPEECH_SECRET` | 한국어 인식·화자 분리. 도메인 생성 후 발급 |
| Azure Speech | `AZURE_SPEECH_KEY`, `AZURE_SPEECH_ENDPOINT` (`AZURE_SPEECH_API_VERSION` 기본 2025-10-15) | fast transcription, ko-KR |
| OpenAI | `OPENAI_API_KEY` (`OPENAI_STT_MODEL` 기본 whisper-1) | segment 타임스탬프는 whisper-1 보장 |
| 기본 프로바이더 | `STT_PROVIDER` = clova \| azure \| openai | 미설정 시 사용 가능한 것 자동 선택 |

키가 하나도 없어도: 자막 생성·감사·샘플 체험은 전부 동작합니다.

## 8. 정직한 한계

- **STT 정확도는 프로바이더에 종속**됩니다. 서버는 전사 결과를 교정하지 않으며(내용 불변),
  잘못 인식된 어휘의 수정은 에이전트·사용자와의 확인 과정에 맡깁니다.
- **`plain_text` 경로의 타이밍은 근사**입니다 (문장 가중치 비례 배분). 응답에 근사임을 명시하며,
  정밀 동기화가 필요하면 타임스탬프가 있는 `segments`를 쓰도록 안내합니다.
- **읽기 속도 위반의 최종 해결(축약)은 서버 밖**입니다. 서버는 간격 차용으로 노출을 연장하는
  것까지만 하고, 그래도 초과하면 목표 자수와 함께 위반으로 보고합니다.
- **오디오는 공개 URL로만** 받습니다 (25MB 이하). 채팅 첨부 파일 직접 수신은 클라이언트
  구현에 따라 다르므로 신뢰하지 않습니다.

## 9. 실행

```bash
npm ci && npm run build && npm start   # :8080/mcp (streamable HTTP, stateless)
npm run demo                            # 키·네트워크 없이 30초 셀프 데모 (임시 포트)
npm test                                # 스모크 13종(서버 기동 필요) + C-1 회귀 + property 2000케이스
```

## 10. KC 배포

Git 소스 빌드(저장소 루트에 Dockerfile) 또는 이미지 등록(`--platform linux/amd64` 필수).
Endpoint: `https://<서버명>.playmcp-endpoint.kakaocloud.io/mcp`
STT env는 KC 컨테이너 환경 변수로 주입 (없어도 서버는 정상 기동).

## 11. 로드맵

- **CLOVA Speech 정식 연동 상시화** — 사업자 계정 확보 후 기본 프로바이더 전환 (국내 기술 우선).
- **Kakao Tools 연계 (본선 ~8/27)** — 생성된 자막 파일을 카카오톡 "나에게 보내기"로 전달하는 흐름.
- **실시간 자막 클라이언트 앱과의 분업** — 실시간 처리는 별도 클라이언트(웹 Notifications/Vibration
  API 영역)가 맡고, 캡션잇은 규격 룰북·링킷은 소통 프로필을 공급하는 구조 (두 서버의 로드맵 합류점).
- **용어집 연동** — 전문용어가 많은 콘텐츠에서 용어 큐 하이라이트 (링킷 축 D 근거).
- **규격 룰북 확장** — KBS 한국어 자막 제작 규정 등 방송사 실무 기준의 수치화·출처 병기.
